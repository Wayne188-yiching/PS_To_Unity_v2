流程規範（每次優化必做）

  1. 版本號：每完成一個階段即 bump 版本號。行為變更或 JSON 格式新增欄位 → 升 minor（如 2.9.0）；純修正 → 升 patch（如 2.9.1）。Phase 1 起版本已改由 version.json + bump-version.ps1 單一來源同步全部位置（.\bump-version.ps1 -NewVersion X.Y.Z；加 -Check 僅檢查不修改）。
  2. HTML 文件：每階段完成後在 GUIDE_zh.html 更新對應段落與 FAQ，並記錄 changelog。任何未來優化的 Phase 一律新增章節寫入本檔（OPTIMIZATION_PLAN_zh.html），不再另建獨立 markdown 規劃檔。
  3. 驗收：每階段附「驗收標準」，由使用者在 Photoshop / Unity 實機驗證通過後才進入下一階段。
  4. 版本控管：每階段結束一律 git commit + git push（跨機器工作流程；Phase 0 起已連上 origin/main）。
  5. Phase 交接提示詞（必附）：每個 Phase 章節末尾都必須放一段可直接複製的「交接提示詞」（<details class="handoff"> 區塊），讓使用者能在任何新 Claude 對話中複製貼上、無需重新解釋背景就能接手該 Phase 的規劃或實作。提示詞至少包含：任務目標、關鍵入口檔案路徑、參考的既有決議段落錨點（如 OPTIMIZATION_PLAN_zh.html#phase4-q3）、以及本次要交接的具體步驟。未附提示詞的 Phase 章節視為未完成規劃。

字型精度診斷（核心痛點）

使用者回報：「文字大小與內外框（描邊）粗細一直對不準」。程式碼追查後確認以下 5 個誤差來源：

#誤差來源位置說明
F1 字級 pt / px 單位混用 已排除 JSX readFontSize() ↔ TmpMapper.Apply() 確認專案 PSD 慣用 72 DPI，此條件下 1 pt = 1 px，程式碼換算結果一致，字級問題不在這裡。若未來設計稿改用其他 DPI 再重新評估。
F2 描邊換算分母錯誤（Padding vs gradientScale） TmpMapper.ConvertPhotoshopStrokeWidth() 實際 Font Asset（GenSenRounded-M SDF）：Sampling Point Size = 25、Padding = 5、SDFAA 模式。
TMP 在 SDFAA 模式下 gradientScale = Padding + 1 = 6，正確公式為：
_OutlineWidth = strokePx × SamplingPointSize / (gradientScale × fontSize)
= strokePx × 25 / (6 × fontSize)　→　ratio ≈ 4.17
但程式碼用 Padding（5） 而非 Padding+1（6），ratio = 5，導致描邊系統性偏厚約 20%。
修正：分母改為 atlasPadding + 1，或動態讀取 Font Asset 的 faceInfo 計算。
F3 描邊超限被無聲截斷 同上，Clamp(…, 0.01f, 1.0f) SDF 描邊的物理上限受 atlas padding 限制。遊戲 UI 常見 4~8px 粗描邊容易超限，被 clamp 後永遠比 PS 細且無任何提示。應改為超限時於 Validate 報告中明確警告並建議重建 Font Asset 提高 padding。
F4 未讀取描邊位置（外/中/內） JSX frameFX 讀取段 目前只讀 size / color / opacity。PS 描邊位置（Styl：outside / center / inside）影響整體視覺寬度達半個描邊寬。Unity 端可用 _FaceDilate 補償（需實測驗證）。
F5 材質庫比對容差放大誤差 TmpMapper.FindLibraryMaterial() 寬度容差 15%：即使換算正確，重用庫存材質球仍可能再偏 15%。待 F1~F3 修正後收緊容差。
校準基準：建立一個固定的「校準 PSD」（多組字級 × 描邊寬組合），每次調整文字相關程式後匯出比對，作為回歸測試基準。

Q1 — 路線分階段

階段範圍版號狀態
Phase 4 (A)GridLayoutGroup + CanvasGroup（Layout child opts 沿用現狀不動，見 Q7）v2.9.0已完成
Phase 4.5 (B)ScrollRect 完整套件（含 Viewport/Content 階層重組、RectMask2D/Mask、Scrollbar、[SCROLL_V]/[SCROLL_H] 命名觸發）延後決定Phase 4 完成後再開規劃

Why 分階段：ScrollRect 需要重組 prefab 階層，跟 A 風險不同等級，獨立做較乾淨。

Q2 — 訊號源：PS 端是 source of truth

JSX 偵測並寫進 JSON → Unity 端只負責照 JSON 掛 component。任何新欄位先加在 PhotoshopUiNode，JSX 寫入，Unity backend 只讀。

Why：跟現行 layoutType / layoutSpacing / contentSizeFitter 同模式；除錯時看 JSON 就能對得上 prefab。

Q3 — 覆蓋語意：每次 full import 完全照 JSON 蓋過

Full import 完全覆蓋 Grid / CanvasGroup（與既有 ApplyLayoutGroup 行為一致）；reskin flow 完全不觸碰新 component（保留設計師手調空間）。

Why：一致性 > 彈性；source of truth 是 PS。Phase 2 立下的鐵律延續。

Q4 — Grid 觸發：顯式 [GRID] / [GLAYOUT] 命名標籤

不做啟發式偵測。

Why：Grid 誤判成本不對稱（強塞 cell size 會視覺整個錯位）；設計師打標籤成本低、行為可預期。也跟 Phase 4.5 [SCROLL_V] 顯式風格對齊。

Q5 — Grid 參數推導規則

欄位規則
cellSize子節點寬高的 (中位數寬, 中位數高)。避 outlier；同寬同高時等價於 first。
spacing.x同 Y 相鄰節點水平 gap 中位數（沿用 calcLayoutSpacing，只在同 Y 對之間算）。
spacing.y同 X 相鄰節點垂直 gap 中位數（只在同 X 對之間算）。
padding沿用既有 calcLayoutPadding（group bounds vs children bounding-box 差）。
startCorner固定 UpperLeft（hardcode、不進 JSON，見 Q12-d）。
startAxis依實際排列自動判：第一行同 Y 節點數 > 第一列同 X 節點數 → Horizontal，反之 Vertical。
constraintFixedColumnCount，constraintCount = 第一行同 Y 節點數。
childAlignment固定 UpperLeft（hardcode、不進 JSON）。

邊界條件：

子題決議Why
Q5.1 constraintFixedColumnCountFlexible 在 runtime 因 viewport 寬度自動換行，跟 PS 看到的版面對不上
Q5.2 子寬高差異 > 20%JSX 報 warning + 降級成普通 group（不掛 GridLayoutGroup）GridLayoutGroup 強制等大 cell；硬塞會視覺爆掉；降級至少不破壞排版
Q5.3 cellSize outlier 差距 ≤ 20%仍掛 GridLayoutGroup，但報 warning20% 內視為設計意圖等大、PS 圖層只是 anti-alias 飄幾 px；warning 幫設計師抓「以為是 grid 但對齊跑掉」的情境

Q6 — CanvasGroup 觸發：root 自動掛 + 中間層 [CG] / [CANVASGROUP] 顯式

JSON root node 一定掛 CanvasGroup（不需要標籤）；中間層 group 想掛要打標籤。單一標籤，不分動詞（不收 [FADE] / [TOGGLE]，因為 CanvasGroup 是同一個 component，動詞是 runtime 的事）。

Why：99% 場景是「整頁淡入淡出 / loading 遮罩 disable 互動」，root 自動掛省標籤；中間層顯式跟 [GRID] 對齊。

Q6.1 — CanvasGroup 初始值：全部 Unity 預設

alpha=1 / interactable=true / blocksRaycasts=true / ignoreParentGroups=false。不讀 PS group opacity。

Why：Phase 4 範圍 = 掛 component，不做「PS opacity → CanvasGroup.alpha」翻譯（scope creep + 語意衝突：bake 進初始 alpha 會打到 runtime fade 動畫）。

Q7 — Layout child opts：沿用現狀，Phase 4 不動

  • Q7-a 四個 bool（childControlWidth/Height、childForceExpandWidth/Height）→ 全部 false（沿用）
  • Q7-b childAlignment → 沿用三種不一致（H=MiddleLeft、V=UpperCenter、Grid=UpperLeft）

Why：Phase 1–3 寫死跑了三個 phase 零抱怨；統一成 UpperLeft 是 breaking change 又收益低，YAGNI。Phase 4 對既有 H/V Layout 完全不碰。

Q8 — 命名標籤定義

觸發目標標籤別名集
Grid[GRID] / [GLAYOUT]
CanvasGroup[CG] / [CANVASGROUP]
（參照）H Layout[H] / [HLAYOUT]（既有）
（參照）V Layout[V] / [VLAYOUT]（既有）
（參照）Fake thickness[THICK:offsetY:offsetX]（既有）

通則（跟既有 convention 鎖死）：

  • 大小寫不敏感（regex i flag）
  • 位置任意（regex 全域 replace）
  • 多標籤可組合：[GRID][CG] Cards 兩個同時生效（Grid 跟 CanvasGroup 正交）
  • 未知 token（如 typo [GIRD]）→ 靜默放行（見 Q10-d）
  • 架構：先沿用「各自寫 stripXxxTags」，第四個 tag（Phase 4.5 [SCROLL_V]）進來時再 refactor 成統一 parser
