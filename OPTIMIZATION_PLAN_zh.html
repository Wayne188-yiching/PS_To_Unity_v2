流程規範（每次優化必做）

  1. 版本號：每完成一個階段即 bump 版本號。行為變更或 JSON 格式新增欄位 → 升 minor（如 2.5.0）；純修正 → 升 patch（如 2.4.3）。
  2. 需同步更新版本號的位置（Phase 1 完成版本號單一來源之前，以下全部手動同步）：
    • PhotoshopUiPackageExporter.jsx 的 SCRIPT_VERSION
    • PhotoshopUiImporterWindow.cs 的 ToolVersion
    • GUIDE_zh.html：header 標題徽章、頁尾版本
    • README.md 與 README_zh.md 的「目前版本」
  3. HTML 文件：每階段完成後在 GUIDE_zh.html 更新對應段落與 FAQ，並記錄 changelog。
  4. 驗收：每階段附「驗收標準」，由使用者在 Photoshop / Unity 實機驗證通過後才進入下一階段。

字型精度診斷（核心痛點）

使用者回報：「文字大小與內外框（描邊）粗細一直對不準」。程式碼追查後確認以下 5 個誤差來源：

#誤差來源位置說明
F1 字級 pt / px 單位混用 已排除 JSX readFontSize() ↔ TmpMapper.Apply() 確認專案 PSD 慣用 72 DPI，此條件下 1 pt = 1 px，程式碼換算結果一致，字級問題不在這裡。若未來設計稿改用其他 DPI 再重新評估。
F2 描邊換算分母錯誤（Padding vs gradientScale） TmpMapper.ConvertPhotoshopStrokeWidth() 實際 Font Asset（GenSenRounded-M SDF）：Sampling Point Size = 25、Padding = 5、SDFAA 模式。
TMP 在 SDFAA 模式下 gradientScale = Padding + 1 = 6，正確公式為：
_OutlineWidth = strokePx × SamplingPointSize / (gradientScale × fontSize)
= strokePx × 25 / (6 × fontSize)　→　ratio ≈ 4.17
但程式碼用 Padding（5） 而非 Padding+1（6），ratio = 5，導致描邊系統性偏厚約 20%。
修正：分母改為 atlasPadding + 1，或動態讀取 Font Asset 的 faceInfo 計算。
F3 描邊超限被無聲截斷 同上，Clamp(…, 0.01f, 1.0f) SDF 描邊的物理上限受 atlas padding 限制。遊戲 UI 常見 4~8px 粗描邊容易超限，被 clamp 後永遠比 PS 細且無任何提示。應改為超限時於 Validate 報告中明確警告並建議重建 Font Asset 提高 padding。
F4 未讀取描邊位置（外/中/內） JSX frameFX 讀取段 目前只讀 size / color / opacity。PS 描邊位置（Styl：outside / center / inside）影響整體視覺寬度達半個描邊寬。Unity 端可用 _FaceDilate 補償（需實測驗證）。
F5 材質庫比對容差放大誤差 TmpMapper.FindLibraryMaterial() 寬度容差 15%：即使換算正確，重用庫存材質球仍可能再偏 15%。待 F1~F3 修正後收緊容差。
校準基準：建立一個固定的「校準 PSD」（多組字級 × 描邊寬組合），每次調整文字相關程式後匯出比對，作為回歸測試基準。

UI/UX 問題清單

高會造成損失或誤操作

#問題位置修正方向
U1 「執行換皮掃描」名實不符：按鈕叫「掃描」，實際直接覆蓋目標 PNG 且無確認對話框 Unity ExecuteReskin() 改為兩段式：先掃描列出「將覆蓋 N 張 / 缺少 M 張」預覽 → 使用者確認後才執行覆蓋
U2 「Check for Updates」直接關閉匯出對話框，已調好的設定全部丟失 PS 對話框 dialog.close(2) 更新檢查改為不關閉對話框，或檢查後返回原對話框並保留所有欄位值

中增加理解成本、引導失敗

#問題修正方向
U3 Unity 端三顆路徑按鈕語意重疊（套用標準輸出路徑 / 圖片位置跟隨來源 / 建立專案資料夾），需閱讀文件才知道何時按哪顆 收斂為單一路徑：填專案名稱 → 自動偵測資料夾是否存在 → 存在即套用、不存在則詢問一鍵建立
U4 步驟有編號 1~4 但無完成狀態；前置條件未滿足仍可按 Generate，得到錯誤而非引導 前置條件未滿足時按鈕 disable 並標示缺項（如「尚未套用 Package」）
U5 必填項（含文字節點時的 TMP Font Asset）無即時提示，要等 Generate 失敗才知道 套用 Package 當下即可判斷有無文字節點，缺字型時立即標紅該欄位
U6 Validate / 預覽（Dry-run）/ Generate 三鈕等大並排，主要動作無視覺主次，且 Validate 與預覽功能重疊 合併為「檢查」+「Generate Prefab」（主視覺按鈕，內建檢查）
U7 換皮工具（獨立、低頻、具破壞性）插在主匯入流程中間 移至獨立分頁或折疊區塊底部
U8 「參考解析度自動跟隨 Layout」toggle 與「套用 Layout 尺寸」按鈕並存，兩個控制項做一件事 保留 toggle；僅在 toggle 關閉時顯示手動套用按鈕

低細節打磨

#問題修正方向
U9PS 對話框全英文，與 Unity 端及文件（中文）不一致PS 對話框中文化（或雙語）
U10Layout JSON 路徑欄位常駐主畫面，但絕大多數情況跟隨 PNG 資料夾即可收進進階區，主畫面僅留輸出資料夾
U11「Use fast layer duplicate export」暴露內部實作細節，使用者無從判斷自動判斷或收進進階區
U12「Export selected text layers as PNG」依賴開對話框前的選取狀態，勾選當下看不到選了哪些圖層顯示「目前選取：N 個文字圖層（名稱…）」
U13匯出結果只有一個 alert，資訊一閃即逝輸出簡要匯出報告檔（與後續「匯出報告」功能合併）

階段排程

版號修正（2026-06-11 Phase 0 執行時發現）：本計畫原以 v2.4.2 為基準，但倉庫實際狀態為 JSX = 2.4.2、Unity 端 = 2.5.0（SkinTheme 等 v2.5 功能已存在）、README = 2.2.0，版本已漂移。為避免 Unity 端版號倒退，Phase 0 改出 v2.5.1（統一全部位置），後續 Phase 版號順延：Phase 1 → v2.6.0、Phase 2 → v2.7.0、Phase 3 → v2.8.0。

Phase 0緊急安全修正 → v2.5.1（原訂 v2.4.3）已實作，待驗收

範圍小（各約十數行）、風險低、但問題會造成實際損失，不等大版本。

  • U1：換皮改為「掃描預覽 → 確認 → 覆蓋」兩段式
  • U2：Check for Updates 不再關閉匯出對話框 / 保留欄位值

驗收標準：換皮在覆蓋前必出現確認清單；PS 端點更新檢查後返回對話框、原設定仍在。

Phase 1字型精度 + 版本基建 → v2.6.0（原訂 v2.5.0）

針對核心痛點「文字大小與描邊對不準」，並讓之後每次迭代的版本同步成本下降。

  1. F2：描邊換算分母從 atlasPadding 改為 atlasPadding + 1（= gradientScale），修正系統性偏厚 20% 的問題。建議同步改為從 Font Asset 動態讀取，避免未來換字型時又出現同樣問題
  2. F3：換算結果超過 1.0 時不再無聲 clamp，於 Validate / Generate 報告明確警告並附建議 padding 值（以目前字型為例：36pt 文字描邊上限約 8.6px）
  3. F4：JSX 補讀 stroke position；Unity 端以 _FaceDilate 補償居中描邊（實驗項，需實測）
  4. 基建：建立 version.json 單一版本來源 + bump script（改一處自動同步 JSX / CS / GUIDE / README）
  5. 校準 PSD：建立固定字級 × 描邊組合的測試檔，作為回歸基準

驗收標準：用校準 PSD 匯出，於 Unity 疊半透明 PS 截圖比對，72 與 144 DPI 文件的文字大小皆一致；常用描邊寬度視覺誤差肉眼不可辨；超限描邊出現明確警告。

Phase 2Unity 端 UX 重整 → v2.7.0（原訂 v2.6.0）

  • U3：路徑按鈕收斂，以「建立專案資料夾」為主流程（最常用）；「套用標準輸出路徑」與「圖片位置跟隨來源」降為進階折疊選項
  • U4 / U5：步驟完成狀態、按鈕 disable 引導、必填即時標紅
  • U6：Validate 與預覽合併，Generate 做主視覺按鈕
  • U7 / U8：換皮移位、參考解析度控制項簡化
  • F5：收緊材質庫比對容差（依 Phase 1 修正後實測決定數值）

驗收標準：新使用者不看文件可完成「套用 Package → Generate」完整流程；每個錯誤狀態都有明確下一步指引。

Phase 3PS 端打磨 + 還原度 → v2.8.0（原訂 v2.7.0）

  • U9 ~ U12：PS 對話框中文化、選項收納進階區、選取狀態顯示
  • U13 + 匯出報告：總圖量、總大小、超大單圖警告、skipped 清單，輸出可回看的報告檔
  • 像素雜湊去重：以匯出後 PNG 像素雜湊比對，正確撿回 v2.4.2 停用的去重功能（不誤殺同寬不同字的文字圖）
  • 漸層文字：讀取 PS Gradient Overlay 上下色 → TMP colorGradient 四角頂點色

驗收標準：重複底圖正確共用、文字圖不被誤合併；漸層標題文字還原上下色；匯出後產生報告檔。

執行方式

分工模式：規劃討論於 Claude Project；程式實作於 Claude Code（桌面版），指向本機 PS_To_Unity_v2 倉庫；Photoshop / Unity 實機驗收由使用者執行。

  • 本文件放入倉庫根目錄，作為 Claude Code 的執行依據。
  • 一個 Phase 一個對話：依序執行，勿一次跨多階段。
  • 每個 Phase 開始前，先請 Claude Code 說明「打算改哪些檔案與原因」再放行。
  • 每個 Phase 結束：bump 版本 → 更新 GUIDE_zh.html 與 README → git commit → 使用者實機驗收 → 通過後進入下一階段。
