Photoshop → Unity UI 工具 v2.11.0

將 Photoshop 排版好的遊戲 UI 轉成 Unity uGUI + TextMeshPro Prefab。文字圖層保留為可編輯的 TMP 節點，非文字圖層逐一輸出為 PNG Sprite。

安裝 主流程 PS 匯出對話框 圖層命名規則 Layout Group 自動排版 Unity 匯入工具 專案資料夾結構 文字與材質 換皮系統 版本更新 常見問題 版本紀錄

安裝

PS　Photoshop 端

兩種執行方式擇一：

方式 A（推薦）：執行一次安裝器，選單只留一個入口 v2.6.1
  1. Photoshop 執行 File > Scripts > Browse…，選擇倉庫裡的 PhotoshopExporter/InstallPhotoshopPlugin.jsx。
  2. 重啟 Photoshop。

之後選單只有一個 File > Scripts > PS To Unity v2，開啟後是工具面板，所有工具（匯出、自動命名、Spine、校準板、Debug 工具、更新）都從面板進入。工具本體留在倉庫資料夾，更新後不需重新安裝。

方式 B：每次手動瀏覽

在 Photoshop 執行 File > Scripts > Browse…，選擇 PhotoshopExporter/PhotoshopUiPackageExporter.jsx（或任一工具）。

不要把整個 PhotoshopExporter/ 資料夾複製進 Presets\Scripts\——那會讓每支腳本各佔一個選單項（舊版說明的做法）。若你之前這樣裝過，重新執行一次方式 A 的安裝器即可：v2.6.1 起安裝器會自動清掉散落在 Scripts 資料夾裡的工具複本（只刪本工具的已知檔名，不動其他腳本）。

Unity　Unity 端

將整個 Assets/Editor/PhotoshopUiImporter/ 資料夾複製到 Unity 專案的 Assets/Editor/ 底下，Unity 會自動編譯。

完成後從選單開啟：

Tools > Photoshop UI Importer > Importer_v2
前置需求：Unity 專案需安裝 TextMeshPro（Package Manager 搜尋 TextMeshPro 安裝即可）。

主流程

1
整理 PSD

確認圖層為英文命名。如需批次改名，執行 PhotoshopLayerAutoNamer.jsx。

2
PS 匯出

執行 PhotoshopUiPackageExporter.jsx，指定 PNG 資料夾與 Layout JSON 路徑，點 Export。

3
Unity 套用 Package

在 Importer_v2 選擇 Package 資料夾，點「套用 Package」，工具自動找 JSON 與圖片。

4
建立專案資料夾

填寫「專案資料夾名稱」，點「建立專案資料夾」，自動建立子目錄結構並生成 SpriteAtlas。

5
指定字型與材質

至少指定「預設 TMP Font Asset」。有現成字框材質庫時，可填入「TMP 材質球資料夾」。

6
Generate Prefab

點「Validate」確認資料，再點「Generate Prefab」完成輸出。

圖層命名規則

Anchor / Pivot 標記

在圖層名稱中加入標記（不分大小寫），Unity 匯入時自動套用對應的 RectTransform Anchor 與 Pivot：

標記Anchor 效果
ANCHOR_TL / ANCHOR_TOP_LEFT左上角固定
ANCHOR_TR / ANCHOR_TOP_RIGHT右上角固定
ANCHOR_BL / ANCHOR_BOTTOM_LEFT左下角固定
ANCHOR_BR / ANCHOR_BOTTOM_RIGHT右下角固定
ANCHOR_C / ANCHOR_CENTER中心固定
ANCHOR_ML / ANCHOR_LEFT左中固定
ANCHOR_MR / ANCHOR_RIGHT右中固定
ANCHOR_TC / ANCHOR_TOP上中固定
ANCHOR_BC / ANCHOR_BOTTOM下中固定
STRETCH_X / STRETCH_HORIZONTAL水平延展
STRETCH_Y / STRETCH_VERTICAL垂直延展
STRETCH_FULL / STRETCH_BOTH全方向延展

未加任何標記時，工具根據元素在父容器中的相對位置自動推斷 Anchor（靠左 / 靠右 / 置中 / 延展等）。

Pivot 可另行指定（可與 Anchor 標記組合）：

標記效果
PIVOT_TL左上角 pivot
PIVOT_C / PIVOT_CENTER中心 pivot
PIVOT_BL左下角 pivot（其他方向以此類推）

特殊前綴與標記

前綴 / 標記作用
IGNORE_ 或 [ignore]匯出時完全跳過此圖層。
REF_參考用圖層，匯出時跳過。
IMG_圖片前綴（前綴本身會從 PNG 檔名中移除）。
TXT_文字前綴（前綴本身會從節點名稱中移除）。
BTN_ v2.4.3圖片節點以 BTN_ 開頭時（不分大小寫），Unity 匯入後自動在該 GameObject 加上 Button 元件，省去手動加掛步驟。

文字圖層輸出控制

個別圖層可在名稱中指定輸出方式，覆蓋對話框的預設設定：

名稱中含效果
[PNG] [IMAGE] [IMG] TXTIMG_強制輸出為 PNG 圖片
[TMP] [TEXT] TMP_ TXT_強制保留為 TMP 節點

材質球標記（文字圖層）

在文字圖層名稱加上材質標記，指定 Unity 要套用的 TMP 材質球：

TXT_title_[MAT:title_yellow_outline]
TXT_score_[MAT:reward_white_red_outline]

Unity 端 Skin Map 可將 materialToken 對應到不同的 TMP Material Preset。若圖層有 Layer Style 的 Stroke 效果，工具會自動產生 materialToken，無需手動標記。

假厚度標記（文字圖層）

在文字圖層名稱加入 [THICK:N]，N 為向下偏移的像素數，Unity 會自動疊兩層 TMP 模擬立體感：

TXT_title_[THICK:3]
TXT_score_[THICK:2]

上層為原始文字，下層為顏色加深（× 0.55）的陰影層，偏移量對應 PS 設計稿像素。同時支援 X 軸偏移：[THICK:2:1]（格式：[THICK:offsetY:offsetX]）。

v2.4 起 PS 端 JSX 已自動解析 [THICK:N] 與 [THICK:N:M]，名稱中加標記即可，不需手動編輯 JSON。

Layout Group 自動排版 v2.4

圖層群組可透過明確命名標記轉為 Unity 的 HorizontalLayoutGroup 或 VerticalLayoutGroup，並自動套上 ContentSizeFitter。未標記的普通群組會完整保留 Photoshop 座標，不再猜測排版方向。

v2.4.2 起停用自動去重：先前依「相同寬高」判斷重複圖會誤判（譬如同寬度但不同字的文字、同尺寸但不同 icon），故 v2.4.2 起每個圖層各自輸出獨立 PNG。若需要圖片共用，請在 PS 將圖層名稱統一。

偵測規則

v2.10.1 起只接受明確命名標記，避免一般群組因子層剛好同 X/Y 而被錯誤重排：

方式條件結果
命名標記群組名稱含 [H] 或 [HLAYOUT]（不分大小寫）強制水平 layout
命名標記群組名稱含 [V] 或 [VLAYOUT]（不分大小寫）強制垂直 layout
無標記任何一般群組普通群組，不加 Layout Component，保留 PS 座標

實際 PSD 圖層命名範例

情境 A：標記強制（不依賴位置）

Settings_Page/
├── BtnBar_[H]/              ← 強制水平
│   ├── Btn_Home.png
│   ├── Btn_Settings.png
│   └── Btn_Shop.png
└── Menu_List_[V]/           ← 強制垂直
    ├── Menu_Play.png
    ├── Menu_Help.png
    └── Menu_Quit.png

情境 B：結合其他標記（多標記共存）

TopBar_[H][ANCHOR_TC]/       ← 水平 layout + 上方置中錨點
├── Coin.png
├── Diamond.png
└── Ticket.png

Spacing / Padding 自動計算

  • Spacing：依子層排序後，計算相鄰子層的像素間距，取中位數作為 spacing。
  • Padding：群組邊界與最外層子層邊界的距離，分別套用到 Left / Right / Top / Bottom。
  • 所有值由 PS 設計稿座標自動算出，不需手動輸入。

重複圖去重（v2.4.2 已停用）

v2.4.0 ~ v2.4.1 曾以「相同 width × height」為判斷依據自動共用 imagePath。但實務上此判斷不準確：

  • 同寬度但不同字的文字節點被誤判為同一張圖。
  • 共用後也會干擾 Layout Spacing 中位數計算，造成排版錯誤。

v2.4.2 起取消自動去重，每個圖層各自輸出一張 PNG。若多個圖層真的指向同一張圖，請將圖層名稱命名一致（如 btn_bg + 副本），工具會以 btn_bg_002、btn_bg_003 的方式分別匯出，仍可由設計師後續手動合併或在 Unity 端用相同 Sprite。

SpriteAtlas 自動設定 v2.4.3：建立後 SpriteAtlas 自動套用專案標準設定—— Packing（無旋轉、無緊密排列、Padding 4）、 Texture（sRGB、Bilinear、無 Mip Maps）、 Android / iOS 平台（ASTC 6×6、2048、Best）。 Objects for Packing 已指向 SpriteAtlas/ 資料夾，放入圖片後在 Unity 點「Pack Preview」即可打包。

Unity 匯入工具（Importer_v2）

開啟路徑：Tools > Photoshop UI Importer > Importer_v2

1. Photoshop UI Package

選擇 Photoshop 匯出的 Package 根資料夾，點「套用 Package」。工具自動掃描找出 layout JSON（優先找 layout.json，其次找 *_layout.json）以及 PNG 圖片資料夾。按「只驗證資料」可在不生成 Prefab 的情況下確認資料正確。

2. Unity 生成位置

填寫「專案資料夾名稱」（例如 Setting_Page），有兩個按鈕可用：

  • 「建立專案資料夾」（新專案建議用）：自動建立完整子目錄結構、生成 SpriteAtlas，並填入輸出路徑。詳見 專案資料夾結構。
  • 「套用標準輸出路徑」（既有資料夾用）：只填入 Atlas 與 Prefab 路徑，不建立資料夾。

圖片來源若已在 Unity 專案 Assets 底下，工具直接使用來源路徑，不重複複製。「參考解析度」預設自動跟隨 Layout JSON 中的 Canvas 尺寸。

3. 文字、材質與換皮

欄位用途
預設 TMP Font Asset所有文字節點的 fallback 字型。UI 含文字時必填。
預設 TMP 材質球文字的 fallback 材質。Stroke 效果自動衍生的材質也以此為基底，建議填寫。
TMP 材質球資料夾（選填）指定一個存放現成字框材質球的資料夾。Generate 時優先從此資料夾比對，找到相近的直接重用，找不到才自動新建。設定後視窗會即時顯示找到幾顆。
Skin Map（選填）將特定 Sprite 名稱替換為換皮版本。
換皮工具（舊版）：可指定美術來源資料夾與 Unity 目標資料夾。v2.5.1 起改為兩段式：先「掃描換皮（預覽）」列出將覆蓋與缺少的清單（不修改任何檔案），確認清單後再按「確認覆蓋」並通過確認對話框，才會真正覆蓋同名 PNG。找不到對應美術的檔名會列出成待辦清單。
SkinTheme 批次換皮（v2.5）：可跨多個 Prefab 批次替換 Sprite 參照，詳見 換皮系統。

4. 執行

  • Validate：讀取 Layout JSON，驗證節點數量與資料格式。
  • 預覽（Dry-run）：顯示節點數、是否含文字、圖片資料夾狀態與預計輸出路徑，不執行任何操作。
  • Generate Prefab：匯入圖片、生成 Prefab，完成後在 Project 視窗自動選取並 ping 出 Prefab。

文字與材質球

TMP 文字節點自動流程

  1. PS 文字圖層的 Layer Style → Stroke 效果，匯出 JSON 時寫出 outlineColor、outlineWidth 與自動 materialToken。
  2. PS 文字圖層的 Layer Style → Color Overlay 效果，填色顏色寫入 color 欄位，Unity 端套用至 TMP 頂點色。
  3. Unity Generate 時，依序執行材質球解析：
    1. 若設定了「TMP 材質球資料夾」，先掃描資料夾內所有材質球，以 Outline 顏色（RGB 各差 < 8%）為第一條件過濾，再從符合色條件的材質球中選 Thickness 最接近的直接重用，不新增。v2.4.3 改為顏色優先比對，確保不同字號的同色外框都能穩定找到同一顆材質球。
    2. 未找到匹配 → 從預設材質球複製一份，套用 _OutlineColor / _OutlineWidth，儲存至 Assets/Temp/{名稱}/Font/GeneratedMaterials/。
    3. 無 Stroke 資料的文字節點直接套用預設材質球，不做任何處理。
材質球重用邏輯：同一頁面若多個文字用了接近的字框顏色，工具會自動共用同一顆材質球，避免 GeneratedMaterials 資料夾膨脹。

描邊寬度換算 v2.6

PS 描邊 px 換算為 TMP _OutlineWidth（0~1）的公式：

_OutlineWidth = strokePx × samplingPointSize ÷ (atlasPadding + 1) ÷ fontSize
  • samplingPointSize 與 atlasPadding 自動從你指定的 TMP Font Asset 讀取，換字型不需改任何設定。
  • v2.6.0 修正：分母改用 atlasPadding + 1（TMP 的 gradientScale），修正先前描邊系統性偏厚約 20% 的問題。
描邊超限警告 v2.6：SDF 描邊有物理上限（受 atlasPadding 限制）。換算結果超過 1.0 時，Generate 完成後狀態列會顯示超限筆數，Console 列出各節點的描邊 px、目前上限與建議的 atlasPadding。依建議值重建 Font Asset（Window → TextMeshPro → Font Asset Creator，調高 Padding）後重新 Generate 即可。

描邊厚度補償係數 v2.6.2

Importer_v2 視窗的「3. 文字、材質與換皮」區提供一條 描邊厚度補償係數 滑桿（範圍 0.3 ~ 1.5，預設 1.0），值會即時存入 EditorPrefs 跨 session 保留。

正常情況下保持 1.0 不需動。Phase 1 校準實測：當 Unity TMP 用的字型與字重與 PS 設計稿完全一致（例如 PS 用 SourceHanSansTC-Bold、Unity 也烘同名 SDF）時，物理公式 W = strokePx × sampling / (2 × atlasPadding × fontSize) 直接命中，補償係數 = 1.000 描邊厚度就跟 PS 重合。

什麼情況才需要動補償係數？

  • 換用非思源系列字型（例如圓體、明體、其他廠商 SDF）：不同字型的 antialias falloff 寬度不同，可能視覺上比 PS 偏厚或偏細，這時用校準板回推合適值。
  • PS 與 Unity 故意用不同字型／字重（例如設計稿用 Bold、實際 UI 用 Regular）：補償係數可微調描邊讓比例好看。

校正方法（必要時）：

  1. PS 端執行「Create Calibration PSD」生成 CalibrationBoard.psd，存檔後用 UI Package Exporter 匯出。
  2. Unity Generate 校準板，把 PS 截圖（100% 縮放）丟進 Scene 疊在生成的 UI 上、50% 透明度比對。
  3. Unity 描邊比 PS 偏厚 → 補償係數調低；偏細 → 調高。二分法逼近直到肉眼難辨差異。
  4. 確定後不需再動，所有後續 Generate 自動套用。

係數作用點是公式末端的純倍乘，不影響超限警告的判斷（仍以原始物理寬度為準）。

手動指定材質球

PS 圖層名稱：TXT_title_[MAT:title_yellow_outline]
            TXT_score_[MAT:reward_white_red_outline]

在圖層名稱加上 [MAT:名稱] 標記，Unity 端會優先以此 token 為鍵，從材質球資料夾或 GeneratedMaterials 中尋找對應材質。

換皮系統 v2.5

換皮系統提供兩種模式，視情況搭配使用：

模式 A：同名 PNG 覆蓋（舊版，單資料夾）

指定「美術來源資料夾」與「Unity 目標資料夾」，分兩段執行 v2.5.1：

  1. 點「掃描換皮（預覽，不會修改檔案）」：工具比對兩個資料夾，列出「將覆蓋 N 張 / 缺少 M 張」與完整檔名清單，此時不修改任何檔案。
  2. 確認清單無誤後點「確認覆蓋 N 張 PNG」，並在確認對話框按「確定覆蓋」，工具才把新 PNG 覆蓋到 Unity 目標資料夾裡同名的舊 PNG。

適合單一資料夾、檔名相同的換皮情境。找不到對應檔名的圖片會列成待辦清單。掃描後若更改任一資料夾路徑，舊掃描結果會自動作廢，需重新掃描。

此模式只能覆蓋「同名檔案」，若新舊圖檔名不同，或 Sprite 分散在多個 Atlas 裡，請改用 SkinTheme 批次換皮。

模式 B：SkinTheme 批次換皮（推薦）

適合「一套新美術，套用到 10～20 個 Prefab」的情境。以「舊 Sprite → 新 Sprite」為單位定義對應表，工具掃描指定資料夾下所有 Prefab，自動替換符合的 Image.sprite 參照，完全不動任何 Script / Button / Animator / RectTransform。

SkinTheme 使用流程

  1. 在 Project 視窗目標位置 右鍵 → Create → Photoshop UI Importer → Skin Theme，建立 SkinTheme.asset。
  2. 在 Inspector 填入：
    • Target Prefab Folder Asset：將含有目標 Prefab 的資料夾從 Project 視窗拖入此欄位。
    • Entries：每一列填入「舊 Sprite（現在 Prefab 用的）」與「新 Sprite（新美術）」，可加多列。
    • Excluded Prefabs（選填）：不想被換皮的 Prefab 加入此清單。
  3. 在 Importer_v2 面板 → 換皮工具 → 選 SkinTheme → 點「套用換皮到所有 Prefab」。
  4. 確認 Dialog 後工具逐一掃描 Prefab，結果顯示「X 個 Prefab 更新，共替換 Y 個 Sprite」。
注意：套用換皮會直接修改 .prefab 檔案，操作前建議先用 Git commit 備份，或確認可復原。

常見問題

匯出後 PNG 有些圖沒有輸出？

可能原因：圖層為 Clipping Mask、Adjustment Layer（色調、曲線等）、或 Ignore/REF 前綴圖層，這些均會被跳過。執行後的 alert 會顯示 skipped 數量。

同一個圖層名稱重複出現，PNG 有 _002 後綴？

工具會自動去重命名，重複出現的名稱加上三位數後綴（layer_002、layer_003…），這是正常行為。建議在 PSD 中保持圖層名稱唯一。

Generate Prefab 提示「請先指定預設 TMP Font Asset」？

Layout 中含文字節點（type = text）時，至少要在 Unity 端指定一個 TMP Font Asset 作為 fallback，工具才能繼續。

匯入後圖片位置跑偏？

確認「參考解析度」與 Canvas 尺寸一致，工具預設自動讀取 layout.json 中的 Canvas 設定，一般無需手動調整。若有自訂 Canvas Scaler，需確認 Match Width or Height 設定與 PS 設計稿一致。

圖片來源已在 Assets 內，不想重複複製？

直接選擇 Assets 內的資料夾作為 Package 資料夾，工具偵測到圖片已在專案內時，會直接使用原路徑，不重複複製。

PS 端更新後腳本空白或損壞？

可能是網路中斷或 PowerShell 執行被防毒軟體阻擋。請至 GitHub 手動下載最新 JSX：
https://github.com/Wayne188-yiching/PS_To_Unity_v2

Unity 端更新顯示「無法解析 GitHub 回應」？

請確認電腦可連線至 api.github.com。部分企業網路會封鎖 API 呼叫，可手動從 GitHub 下載 zip 並覆蓋 Assets/Editor/PhotoshopUiImporter/。

點「建立專案資料夾」後 SpriteAtlas 沒出現？

確認已填寫「專案資料夾名稱」再按按鈕。建立後若 Project 視窗未更新，執行 Assets → Refresh（或按 Ctrl+R）即可看到 SpriteAtlas.spriteatlas。

Generate 後文字顏色不正確？

工具讀取 PS Color Overlay 效果的顏色作為 TMP 頂點色。若文字使用的是 Gradient Overlay（漸層），目前版本僅支援純色，漸層上下色需等待後續版本支援。

材質球資料夾設定後，Generate 還是一直新建材質球？

v2.4.3 起改為顏色優先比對：先以 RGB 各差 < 8% 篩選候選材質球，再從候選中選 Thickness 最接近的。若庫中沒有任何材質球顏色符合（差異超出 8%），工具才會自動新建。可確認庫中材質球的 _OutlineColor 是否與 PS 設定的 Stroke 色相近。

SkinTheme 套用後，Prefab 的 Script / Button 還在嗎？

是，SkinTheme 只修改 Image.sprite 參照，完全不動 Script 序列化欄位、Button onClick 事件、Animator、RectTransform 等任何其他元件。

Unity 更新時出現 403 速率限制錯誤？

v2.4.3 起已修正：Unity 端更新不再呼叫 GitHub Contents API（舊版會觸發每小時 60 次的速率上限），改為直接從 raw.githubusercontent.com 下載已知清單，不受速率限制。若使用舊版工具遇到 403，請先手動複製最新 PhotoshopUiImporterWindow.cs 到 Unity，再執行一次更新即可。

文字在 PS 顯示 36pt，Unity 卻變成 29pt？v2.4.1 已修正

v2.4.0 之前使用 textItem.size，這個 API 拿到的是「設計時原始字體」，不含 Free Transform 縮放。若文字圖層被自由變形縮放過，PS 字元面板顯示的是「縮放後的有效字體」（impliedFontSize），兩者會不同。
v2.4.1 起改用 ActionDescriptor 讀取 impliedFontSize，與 PS 字元面板顯示完全一致。

文字為什麼變成圖片？v2.10.1

所有字型預設都會輸出為 TMP。只有文字圖層名稱含 [PNG] / TXTIMG_ 等圖片標記、被選為「目前選取的文字圖層為 PNG」，或使用者明確勾選「白名單外字型改為 PNG」時才會烘成圖片。多字型 PSD 請在 Unity 使用 TmpFontMap 將 fontToken 對應到正確的 TMP Font Asset。

如何建立 Layout Group？v2.10.1

請在群組名稱加入 [H]、[V] 或 [GRID]。v2.10.1 起不再依子層 X/Y 自動猜測，避免 Unity 重排未標記的設計稿群組。

v2.9.1（2026-07-07）— Phase 4 hotfix：Grid 90° 翻轉 + sibling 填格順序 + 警告重複

  • constraintCount 邏輯反了（Grid 90° 翻轉）：v2.9.0 的 calcGridParams 用 horizontal ? firstRowCount : firstColCount 計算 constraintCount。但 Unity Constraint.FixedColumnCount 的語意始終是「欄數」，跟 startAxis 無關；startAxis 只控 GridLayoutGroup 填格順序（Horizontal→row-major、Vertical→column-major），不改變欄數定義。原邏輯會讓 rows > cols 的 grid（例如 1 欄 × N 列的縱向清單）被誤設 constraintCount = N（column 數），視覺整個 90° 翻轉。修正為固定 constraintCount = firstRowCount。
  • 新增 sortGridChildren：sibling 填格順序對得上 PS 視覺：Unity GridLayoutGroup 依 Transform sibling index 依序填格，但 PS 端 group 內圖層順序（PS 排序 = 深度）跟視覺 x/y 位置不一定一致（設計師手動調過圖層深度時尤其明顯）。v2.9.0 直接把 PS 順序寫進 node.children，Unity 端會出現「格子順序跟 PS 看到的不同」。新增 sortGridChildren：依 gridParams.startAxis 對應的填充順序重排 children——Horizontal 排成 row-major（row0 left→right，然後 row1，…），Vertical 排成 column-major（col0 top→bottom，然後 col1，…），跟 Unity 實際填格順序一致。
  • pushWarning 加 (node, code) dedup：applyLayoutGroupMetadata 會在 createGroupNode（第一次算 children/bounds）與 refreshGroupBounds（重新算 bounds 之後）各執行一次，同一個 grid 的 GRID_OUTLIER / GRID_DEGRADED 會被 push 兩次，Unity Console 出現雙份警告。pushWarning 內用 (nodeName, code) 組 key 記在 context._warningKeys，重複的直接跳過。

v2.9.0（2026-07-01）— 優化計畫 Phase 4：自適應布局 Component 自動掛載

  • GridLayoutGroup 自動掛載：PS 端 group 名稱含 [GRID] 或 [GLAYOUT] 標籤時，Unity 端 prefab 自動掛 GridLayoutGroup。cellSize 取子節點寬高中位數、spacing 取同 Y / 同 X 相鄰間距中位數、constraint=FixedColumnCount、constraintCount = 第一行同 Y 節點數、startAxis 依實際排列自動判、startCorner / childAlignment 固定 UpperLeft。子節點寬高差 > 20% 會走 GRID_DEGRADED warning 降級成普通 group（避免 Grid 強制等大 cell 弄爛設計稿排版）；差 ≤ 20% 但有 outlier 則走 GRID_OUTLIER warning 仍掛 Grid。
  • CanvasGroup 自動掛載：prefab root 一律掛 CanvasGroup（99% 場景需要「整頁淡入淡出 / loading 遮罩 disable 互動」，省一個標籤）；中間層 group 想掛要標 [CG] 或 [CANVASGROUP]。初始值全走 Unity 預設（alpha=1、interactable=true、blocksRaycasts=true、不讀 PS group opacity）。
  • 多標籤可組合：[GRID][CG] Cards 兩個同時生效（Grid 跟 CanvasGroup 正交）；大小寫不敏感、位置任意；未知 token 靜默放行（PS 圖層命名習慣不強加規則）。
  • Warnings 統一 channel：JSX 端把警告寫進 layout.json root-level warnings: [{node, code, message}, ...]，Unity 端 LogLayoutWarnings 於「檢查 Layout」/「Validate」/「Generate」三個入口都會 dump 到 Console。code 用 SCREAMING_SNAKE 常數（GRID_DEGRADED / GRID_OUTLIER / UNITY_TOOL_OUTDATED）。
  • Schema 2.8 → 2.9：Unity 端遇到 schemaVersion 比自己認得的還新 → 加 UNITY_TOOL_OUTDATED warning 提示更新工具，防「設計師標了 [GRID] 但沒生效」的悶燒 bug。老 JSON 缺新欄位 → 走預設（不掛 component）。
  • Reskin guard：PsUiSkinApplier.cs 開頭加註解「reskin flow 不可碰 CanvasGroup、GridLayoutGroup、layoutType=grid 的 group 之 layout 欄位」——這些是 full import 的 source of truth，reskin 只換材質與 sprite。
  • 規劃決議整併：Phase 4 grill-me Q1–Q13 完整決議搬進 OPTIMIZATION_PLAN_zh.html#phase4-decisions，原 docs/PHASE4_PLAN.md 廢除；程式碼註解引用一併換成 HTML 錨點。

v2.8.1（2026-06-16）— Phase 3 hotfix：補上 Unity 端 pixel-content 去重

  • 問題：v2.8.0 在 JSX 端對 PNG bytes 算 FNV hash 做去重，但 PS ExportOptionsSaveForWeb 對相同像素內容產生的 PNG bytes 並不穩定（內嵌資訊與壓縮策略差異），導致視覺一模一樣的圖（例如清單列表 4 列共用同一張底圖）hash 完全不同，dedup 沒抓到，atlas 內出現多份重複 PNG。實測一張 lobby UI Package：43 張 PNG 內 7 個視覺重複群組共 12 張可去重。
  • 修正：把第二道去重搬到 Unity 端 ImageImportService。所有 PNG 匯入後新增 pass：用 Texture2D.LoadImage(bytes) 解碼到 in-memory texture，對 GetRawTextureData() 算 MD5（含維度作為 key），同 hash 視為視覺相同 → 保留第一張、刪除重複的 .png 與 .meta、sprites 字典所有 mapping 改指 canonical sprite。
  • 狀態：JSX 端 v2.8.0 的 FNV byte-hash dedup 保留作為第一道（零成本捕捉真正 byte-identical 的情況）；Unity 端 pixel-hash 是第二道，捕捉 byte 差但 pixel 同。Generate 完成後 HelpBox 顯示「像素去重合併：N 張（省 X KB）」。

v2.7.2（2026-06-15）— Phase 2 hotfix：TMP 對齊 fallback 改為 Center

  • 問題：PS Point Text 圖層 bbox（寫進 layout.json 的 width/height）通常比實際字形外輪廓寬，TMP RectTransform 直接套這個 bbox 後，若 alignment 用原本的 Left fallback，字形會貼在 bbox 左邊緣。配上同按鈕內獨立的 icon 圖層（icon 走自己的 X 座標），視覺上會出現「文字 + 假空隙 + icon」的左右順序錯覺（例如「跳過」按鈕的 ▶| 看起來跑到右邊）。
  • 修正：TmpMapper.Apply 把 ParseAlignment 的 fallback 從 TextAlignmentOptions.Left 改為 TextAlignmentOptions.Center。exporter 有明確寫 alignment 欄位時仍以該值為準，只影響「沒帶對齊資訊」的情況，對 PS Point Text 短按鈕標題是更貼近設計稿的選擇。
  • 長期：根治需 JSX exporter 把 width/height 改寫為「字形實際外輪廓」而非「PS layer bbox」，並補上 alignment 欄位。已排程到 Phase 3。

v2.6.3（2026-06-14）— PS 端關閉自動 anchor 推測（hotfix）

  • 問題：先前 inferLayout 會依圖層 bbox 自動推測 anchor（fixed / stretch / 角落）並寫進 layout.json。實測在含群組 effect 的 PSD 上推測結果常常錯誤，會把 331×246 的圖層誤認成 stretch_full、把固定位置的圖層 pivot 標成 (0.5, 0.5) 但 anchor 標成角落，importer 端寫進 RectTransform 之後位置全錯。
  • 修正：inferLayout 一律回傳 center fixed（anchor = pivot = (0.5, 0.5)），所有沒有命名標記的圖層都改用「中心固定 + 原寸 sizeDelta」放置。命名標記（ANCHOR_TL / STRETCH_X / PIVOT_* 等）的 opt-in 響應式行為不受影響，仍由 parseLayoutTag 處理。
  • 影響：所有現有 PSD 需重新執行 UI Package Exporter 才會套用新規則。Unity 端排版會與 PS 設計稿 1:1 對齊；若 Canvas referenceResolution 與 PSD canvas 不一致仍會在 Game View 邊緣裁切，建議把 CanvasScaler 的 referenceResolution 設成與 PSD canvas 相同（例如 2340×1080），或在需要響應式時於 PS 圖層加 ANCHOR_* 命名標記。
  • 已知未修：群組 bounds 受 PS Layer Effect（陰影 / 描邊）影響時會超出父群組或 canvas，需後續單獨修。中心固定模式下子節點位置不受影響，先不動。
